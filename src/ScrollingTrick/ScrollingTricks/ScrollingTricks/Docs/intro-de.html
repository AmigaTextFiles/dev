Dieses Dokument beschreibt die besten Scrolling Algorithmen für Spiele, deren Level auf Blöcken (im Englischen oft auch 'tiles' genannt) basieren. Einige der bekanntesten Spiele dieser Art sind Turrican, Lionheart, Giana Sisters, Z-Out, Katakis, R-Type, Battle Squadron.

Der Hauptgrund dafür, daß Level nicht aus einer einzigen großen Grafik bestehen (sollten), ist, daß dies eine Unmenge an Speicher verschlingen würde und man deshalb nur sehr kleine Levels haben dürfte. Wenn man die Level jedoch aus Blöcken, deren Größe meist 16 x 16 oder 32 x 32 Pixel ist, zusammenbaut, dann kommt man mit ungleich weniger Speicher aus. Dies ist deshalb so, weil man normalerweise mit einer begrenzten Anzahl von Blöcken auskommt und der Speicherverbrauch pro Levelblock (d. h. für einen BLOCKBREITE * BLOCKHOEHE Pixel großen Bereich, wie ihn der Spieler auf dem Bildschirm sieht) nur wenige Bytes (Minimum 1) beträgt.

Die Techniken, die ich hier beschreiben werde, sind für das Amiga Chipset (ECS/OCS/AGA) ausgelegt. Eine große Rolle spielt dabei der Copper, der es ermöglicht, jedes erdenkliche Scrolling (nur horizontal, nur vertikal, oder 8-Wege-Scroller) so zu programmieren, daß es nie notwendig ist, tatsächliche Grafik mit dem Blitter oder der CPU zu scrollen, d. h. im Speicher hin- und herzuschieben. Es reicht stets aus, nur die Blöcke, die "neu hereinkommen" zu blitten.

Soweit ich weiß haben VGA Grafik Chips, also auch Amiga Grafikkarten, alle zum Scrollen notwendigen Features standardmäßig eingebaut:

  • Möglichkeit, die 'display start address' zu ändern
  • Möglichkeit, die zu überspringenden Bytes, um von einer Scanline in die nächste zu gelangen (Modulus), zu ändern
  • Eventuell die Möglichkeit, das Display zu splitten, d. h. nach einer bestimmten Anzahl von dargestellten Zeilen, die 'display start address' mitten im sichtbaren Bereich erneut zu ändern
Deshalb ist es gut möglich, daß die meisten Scrolling Algorithmen in sehr ähnlicher Form auch auf Grafikkarten funktionieren, wobei eventuelle Änderungen wohl hauptsächlich aufgrund des unterschiedlichen Grafikformats (Chunky Pixel statt Planar) notwendig sind. Eine andere Frage ist natürlich, inwieweit die oben genannten Features von der RTG Software bzw. dessen API dem Programmierer zur Verfügung gestellt werden.
