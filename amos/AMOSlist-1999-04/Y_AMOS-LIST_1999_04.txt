Here's the trick:

A=Gui Menu(type)
1 - Menu number.

was selected. A=Gui Menu(2) will tell you which item in the menu, and so on.

"Alastair Murray" <Alastair.Murray@ukgateway.net>


>Let me get this straight. If I went out and bought a standard PC monitor
and
>an external scandoubler/flickerfixer, I would not be able to use Amos.
>Right or Wrong?


Rough description:

Because AMOS does not run through the operating system, it's screen output
will ONLY go to the native AMIGA video outputs.
Most standard screen modes (including all the ones AMOS uses) coming from
the native video output can not be displayed on a monitor, only a TV.
Therefore you must use a scandoubler to view this output on a monitor.

If you have a graphics card, you plug your monitor into the card and have
nothing plugged into your native video output.
Programs have to be told via the operating system to retarget their screens
to go through the graphics card instead of the native video outputs.
AMOS doesn't make screens through the operating systems therefore AMOS will
ALWAYS output its display to the native video outputs.
Meaning if you have a graphics card (with no extra features like
passthroughs, monitor switches etc.) then you have nothing plugged into the
port where AMOS sends it`s signals too.
Therefore you cannot see any AMOS screens.

That is the situation a lot of people are in as you can't use an AMIGA very
seriously now a days without a graphics card.
Therefore AMOS has rapidly become practically useless if you want to
distribute your products at all.

The GUI extension helps a bit but limits what can be done.

On 09-Apr-99, Leto Kauler wrote:

> I'm having some (well, a fair bit) of trouble with icon pasting on
> double buffered screens.

> I've attempted to use Turbo's "F 16 Icon" but this only seems to
> paste to one plane too! Can anyone help???

You can have an UPDATE variable that tells which icons you have changed, and
then have it drawn twice (one for each screen). Something like this:

Dim _UPDATE(15,15)
Dim _MAP(15,15)

Double Buffer
Autoback 0
Bob Update Off

Do

'Add Powerup
If Rnd(100)<5
X=Rnd(15)
Y=Rnd(15)
_MAP(X,Y)=10

'This tells the draw routine to draw this icon twice
_UPDATE(X,Y)=2
End If

'Draw icons
For Y=0 To 15
For X=0 To 15
If _UPDATE(X,Y)>0
Dec _UPDATE(X,Y)
Paste Icon X,Y,_MAP(X,Y)
End If
Next
Next

Bob Draw
Screen Swap
Wait Vbl
Bob Clear

Loop

0 and manually paste the icons to both planes, ie the Logic and
the Physic screen. The Physical is what the user sees, the logical
screen is what you draw on. It's easiest (and most correct) to think
about yourself having two seperate sceens, one that is being shown to the
user and one that you are writing to. Physical and Logical simple denote
where the two screens are. Think of screen 1 as being shown in the
physical space, the user sees screen one, screen 2 is then in the
Logical, hidden space, your draw commands go there. When you 'Screen
Swap' the two screens switch places, screen 1 goes to the logical,
hidden spot and screen 2 moves to the physical, visible spot.
So start at the start, Screen 1 is in the physical spot, screen 2
is in the logical spot. Now draw your icons, they will be drawn to the
screen in the logical spot, screen 2. Now screen swap, screen 2 appears
before the user with your icons drawn. Screen 1 goes to the logical spot.
Notice though that screen 1 is still totally blank, if you screen swap now
and the user will just see a blank screen. Swap quickly like this and the
icons will flicker like mad on the user's screen. You will litterly be
swapping between a screen where the icons have been drawn and one where
they haven't been drawn.
Ok maybe you needed that refrsher maybe you didn't the point is to
solve your problem you have to be sure and draw to both screens:
something like this:

screen #
Paste_Icons Routine
Screen swap
Wait Vbl
Paste_Icons Routine
Screen Swap
Wait vbl

To do this "in real time" ie w/o a big pause in game-play you'll
have to cordinate this so you paste the icons across two game loops,
each game loop is usually
Draw Stuff
Screen Swap

That will only draw to one screen. To work right you'll have to to
paste them to screen 1 on one loop and then again paste them to the screen
2 on the next loop. Make sense?

One way to speed up your code instantly is to split up your code so that
it executes over several cycles.
For example...
1. First cycle - scan player pos / figure out where player is heading
2. Second cycle - do nothing
3. Third cycle - calculate firing / attack positions and execute
4. Fourth cycle - do nothing
5. Fifth cycle - finish AI calculation and execute
6. Sixth cycle - do nothing.
This will speed up the processing of the game and even though the AI is
running at about 1/6th the speed, because the frame update is so fast,
and the player can only react at a certain speed you will get the
illusion that the computer is reacting every cycle.
You will have to keep missiles moving and enemy moving along their old
paths during the 'do nothing' cycles.

Hehe :) Oooppsss... it doesn't works :() because the integer gadget use the
same justify method, and so the number is always shifted to the left. No
problem.... see below...

> When the Integer gadget is selected in the gui it calls the following
> code Where G=Gui Wait
>
> If G=37
> _STEP_SEQ_VELOCITY_1(_SEQUENCE)=Gui Code
> If _STEP_SEQ_VELOCITY_1(_SEQUENCE)>127
> Gui Beep
> _STEP_SEQ_VELOCITY_1(_SEQUENCE)=127
> End If
> Gui Set 1,37,0,_STEP_SEQ_VELOCITY_1(_SEQUENCE)
> End if

Oh my God! :) The IF STEPSEQ blah blah >127 control must be removed, and
Gui Range used instead.

The only way to justify again correctly the gadget is this:

Gui Set 1,37,0,0 : Gui Set 1,37,0,STEPSEQ_VELOCITY_1(_SEQUENCE)


But do you have this kind of IF code for every integer gadget of your GUI?? If
so, you can reduce everthings to only few lines of code:

Example: i suppose your window is the window number 1....


G=Gui Wait : GC=Gui Code : WIN=Gui Window


If WIN=1


If G>=0 : Rem Check if the user has clicked a gadget and...

If Gui Kind(WIN,G)=3 : Rem .....checks if the gadget is a INTEGER gadget...

STEPSEQ_VELOCITY_1(_SEQUENCE)=GC : Gui Set WIN,G,0,0 : Gui Set WIN,G,0,GC

EndIf


EndIf


This simple IF handle *ALL* the integer gadgets of your GUI, independently
on wich GUI they are. The 127 limit is automatically applied by a previous
call to Gui Range.

Dinamic code... the secret of good programming.

visit the site from time to time for updates and further news.

telcos, the regulators, and all the other cosy partners in the Great Phone
Waltz that enough is enough! We're fighting back and won't give ground any
more!

Historically, the preferred method for sending binaries with emails has been
uuencoding, which uses a nice, safe, harmless method of converting binary to
ascii, which can then be inserted swiftly and painlessly into an email,
which is supposed to be raw 7 bit ascii. This then, is a doddle to remove
from normal email files, which normally comprise of multiple messages. You
type one command, and there is the file. If you get problems, you save the
message seperately, then uudecode.

Now, base64 and mime. Nggggggggggg. Horrible. Mime is multimedia
encapsulated emails, where each section of the file is broken down into
chunks and mailed in a combined format, with breaks in the mail denoted by a
unique character line, which then needs to be extracted. This means saving
the whole email to a file, loading it up in a text editor (like emacs),
cutting out the huge base64'd binary chunk, creating a new file, yanking
back the binary chunk, saving to disk, exiting, then remembering the correct
way of debase64'ing to another file, renaming the damn thing, then loading
it, and find you missed one sodding character that didn't quite make it at
the end of line 6783, then going back and doing it again. Endoding is just
as bad. You need one seperate way of mime/demiming the document, then
another for enbase64/debase64'ing the actual binary. Completely annoying.
If that sounds overcomplicated, well, don't do it.

Not everyone uses yam and express. Some people much prefer real software
without the slow but flashy gui. Some people hate flipping mails they can't
actually read.

In general, the gui based email things use mime documents, which is why you
often see two messages on the screen, one in html, and one in pure ascii.
These enhancements are extremely annoying if you aren't using the same
software as the sender. Yeah, I suppose you could debase64 the file, and
output to ibrowse, but that seems just silly.

What are you doing????? :))

In order to detect if the mouse is over the gadget, you need the Gui
Check() command. It checks the specified X and Y coordinates to see if a
gadget exists. If a gadget does exist, the number of the gadget is
returned, else -1 is reported.

But before we need to real-time monitor the mouse movements... and so Gui
Mouse Report is the solution.

This is the code: (i consider the window number 1 and the gadget 0)



Gui Open 1,1,20 : Gui Mouse Report 1,True

'The Y Center of the gadaget...
'------------------------------
Y1=Gui Gad Height(1,0)/2+Gui Y Gad(1,0)


Repeat

G=Gui Wait : X=Gui Mouse Ux : Y=Gui Mouse Uy

If G=-11 : Rem Mouse click detected...

GA=Gui Check(1,X,Y)

If GA=0
INVISIBLE_SLIDER=1
Else
INVISIBLE_SLIDER=0
End If

Else If G=-12 : Rem Mouse pointer moved.....

If INVISIBLE_SLIDER : Gui Set 1,0,0,Y1-Y : End If

End If

Until G=-1

Gui Reset


This program doesn't work exactly as you want, because it's not possible to
do it exactly as you want :) The only difference is the mouse key handling.
When you click on the gadget, the invisible slider start moving..... when
you click again somewhere it stop. I won't bother here, anyway due to a
problem with gadtools in this case you can only partially check the mouse key
press.


About the mouse click events.. is important to specify this:

when the user click the mouse, you'll receive *two* -11 events. The first
when the button is held, and the other when it's released. So you can easly
monitor if the mouse button is clicked or not. But in this specific case,
you *DON'T *receive the first -11 because you've clicked a gadget... you
receive just one mouse event (-11) when you release the key, hence you're
not able to check the hold/release action.

Anyway the above program works nicely... i've tested it before this time :)

Oohh... poor little gadtools ;) It's small, but it works fine. I've found
the solution!

*We can obtain the effect exactly as you want!* We have problems with the left
mouse button, because when you click a gadget the gadtools consider the
action as a gadget press and not as a mouse click, even if the gadget is
not "selectable". BUT... we can use the _RIGHT_ mouse button without
problem!! If you click a gadget using the right mouse button, the action is
not considered a gadget selection, but a standard mouse click, and so we
can easly monitor the mouse key hold/release position and obtain the
invisible slider effect. The secret is the Gui Rmb command.

Here the code: (window 1 - gadget 0 )


Gui Open 1,1,20 : Gui Rmb 1,0 : Gui Mouse Report 1,True

'The Y Center of the gadget
'--------------------
Y1=Gui Gad Height(1,0)/2+Gui Y Gad(1,0)

Repeat

G=Gui Wait : X=Gui Mouse Ux : Y=Gui Mouse Uy : CD=Gui Code
GA=Gui Check(1,X,Y)

If G=-11

If CD=105

If GA=0 : SLD=1 : End If

Else

SLD=0

End If

Else If G=-12

If SLD : Gui Set 1,0,0,Y1-Y : End If

End If

Until G=-1

Gui Reset



Try it, and let me know.

Hi

This archive contains several files, all ASCII text
MS-DOS QBasic programs. Some are maths based, some
are sound based. Some of the files may be broken or
require other files, and therefore not work as
included, but the algorithms have been tested and do
work. Some of the files are repeated with slight
modifications, I lost track of which files were the
latest. I hope 22k is not too much for this list, as
a few people have asked me for this stuff and I
forgot exactly who.


HIGHLIGHTS:

The matrix files contain a set of functions for matrix
algebra, using strings as dynamically allocated memory.
This is slow, but it works well.

The state files use matrix algebra and calculus to
form a discrete time version of continuous time
differential equations, which means the output can
be calculated faster and much more accurately.

The invlap files manipulate s-domain rational
functions, using partial fractions to calculate the
impulse response.

The proto files were a test version of SLab, my current
major project (a powerful sound processing system).
Simple commands are typed in to manipulate 8 buffers,
with simple effects, as well as s and z plane filters.


WARNING:

Most of these were written over a year ago, as experiments
rather than complete programs. There is very little
documentation, but if you really want to use parts of this
or want me to explain what I was trying to do, then I'll be
happy to help. Hopefully I'll document more fully in
the future...


OTHER STUFF:

There is virtually no documentation for these programs.
However, I do have some Microsoft Word (ancient version,
before macro viri...) documents that try to explain some
of the maths a bit, with equations. These were written as
a reference for myself, so some parts are brief. If
anyone is interested I can send them (privately). Like
the programs, they are in various states of completion.

I also have a large collection of bits and pieces of code
that draw various fractals, "interesting" curves, and all
sorts of weird maths stuff. All entirely useless, but in
a kinda curious interesting strange way. Again,
if you want them I can dig them out. (NB: I tried to
transfer them from our old 386 to a fat new pentium, but
the multi-file program seems to have collapsed. Basically
the whole program will have to be rewritten (most could
probably be done using search and replace) if it is to be
stable. The bits that I fixed do fly though (what
used to take 5 minutes on a 386 now happens instantly,
which kinda broke my delay loops :)).
