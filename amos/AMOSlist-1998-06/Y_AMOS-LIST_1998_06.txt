1/

2/

P$=Mid$(Str$(Max(Int(P(Gui Read (WNR,0))-P#),0)),2)

Seems like the compiler can't handle more than a

On 08-Jun-98, Declan_Gorman@modusmedia.com wrote:
> How can I make sure that my program works on another persons machine
> without fault?

What I do is make up a bare bones floppy (startup-sequence, devs, c, l, libs,
fonts dirs - all empty) and boot from that. If it works, you're laughing.
Remember to use the preboot menu (both mouse buttons) to shut off your hd so
it won't look for any files there. Also, you can use snoopdos to see all the
files your program is looking for when it runs.

The two things at the bottom that send the ball back and forth are called

Getting the system date in AMOS

Tue Jun 9, 1998 12:59 am
Joona I Palaste <palaste@cc.helsinki.fi[REMOVE.ME.TO.ACCESS.E-MAIL]>


Just for a funny turn, I thought I'd tell you how I can read the system
date in my AMOS games "Mr UFO" and "RoboQuest". AMOS has no standard
command for this. I've heard it can be done with some memory functions,
but I couldn't bother doing this, so I "cheated". Here is what I did:

If Exist("c:date")
  Exec "c:date >t:date.tmp"
  Open In 1,"t:date.tmp"
  A$=Input$(1,Lof(1))
  Print "The system date is: ";A$;"."
  Close 1
Else
  Print "c:date does not exist."
End If

Re: Bobs, colours and backgrounds...

Fri Jun 19, 1998 4:03 am
Bernie Noel <bnoel@club-internet.fr>


Hi,

>My background picture is 32 colours - OK no problem -
>My bobs (the cars) use only the 8 first one. So, to save chip
>mem, I made my bob bank with 8 colours for the bobs.

> BUT... When I put these bobs on the background, their colours
>change when they are displayed on a zone which contains the 32-8
>other colours.

Your bob has 3 bitplanes, your screen has 5. When your bob hits the screen,
it's going to replace the 3 first bitplanes and leave the other 2 unchanged.
That's why your bob's colours are altered (the screen pixels haven't been
completely replaced by the bob's pixels, only their lower layers have).

So you've got to find a way to empty the last 2 bitplanes of the screen, at
the bob's location. To the best of my knowledge, that's no simple task. 'Set
Bob' could have worked, but unfortunately it doesn't. I guess you have to Doke
to Phybase(3) and Phybase(4) or something. Anyway it's going to be horribly
complicated and slow.

I think there are but 2 solutions:

1) Arrange to display your 8-colour bobs only on background drawn using the
first 8 colours of the screen.

2) Draw your bobs in 32 colours... they'll be prettier :-)

On 1.3. you can use directly the Dos.library/AssignLate function with the
=Doscall() Amos function.

   NAME
        AssignLate -- Creates an assignment to a specified path later (V36)

   SYNOPSIS
        success = AssignLate(name,path)
        D0                    D1   D2

        BOOL AssignLate(STRPTR,STRPTR)

   FUNCTION
        Sets up a assignment that is expanded upon the FIRST reference to
the
        name. The path (a string) would be attached to the node. When
        the name is referenced (Open("FOO:xyzzy"...), the string will be
used
        to determine where to set the assign to, and if the directory can
be
        locked, the assign will act from that point on as if it had been
        created by AssignLock().

        A major advantage is assigning things to unmounted volumes, which
        will be requested upon access (useful in startup sequences).

   INPUTS
        name - Name of device to be assigned (without trailing ':')
        path - Name of late assignment to be resolved on the first
reference.

   RESULT
        success - Success/failure indicator of the operation

> How do I check if a bob collides with another bob.

The 186th page of the manual said:
c=BOB COL(n [,s TO e])
The BOB COL function checks bob number n for a collision wwith another bob.
If a collision has been detected, the value returned in c will be set to -1
(true), otherwise it will be loaded with 0 (false).
Normally the command will check for all collision, but you can specify a
collection of bobs to be tested using the optional range parameters s to
e....

> in AMOS, a little Blitz, some assembler and a smidgen of HiSoft BASIC
> 2. So, I am not sure how much of a help the RKMs would be for these
> languages (besides assembler from which I only make small routines).

If you can find a set, they are worth their weight in gold IMHO.
Whatever language you use, underneath it is still the Amiga OS,
and there is no better guide to using it effectively. Just as a
point of reference, a huge number of questions that I see on this
list could easily be answered in detail with the RKM's. Libraries
and Devices portions are the most useful. Hardware is interesting,
but dated, and largely unneeded if you are going for "OS friendly".
The Autodocs is probably not terribly useful, as this has changed
more (and for the beter, I might add) than any other area, and is
probably best obtained via the developer CD.

> Since I have OS3.0 and would probably be doing more and more things
> geared for OS3.x, would the RKMs be helpful? I assume for backward
> compatibility and for commands that do not change. Is there a better
> source for OS3.x RKM like documents? For instance, isn't cd.device a
> 3.x device? Is it explained in the current RKMs?

The majority of 3.x was in place in 2.1, so the older RKM's are
still largely current. Most of the core principals remain constant,
so even a 1.3 Libs and Devs manual can be quite helpful (My printed
copy is 1.3) Earlier than 1.3 is probably pushing it too far.
As for the cd.device, although no RKM was ever issued for OS3 or
beyond, the Amiga OS 'device' abstraction is consistent, and the
explainations of devices within the RKM, coupled with the specifics
for the cd.device, which can be obtained from an electronic copy
of the Autodocs, should be virtually everything one could need.

> What about that developer CD? Is it worth it? I mean I don't plan on
> writing anything commercial or cutting edge. Right now I'd like to
> make a CD player and a small, turn-based internet game. Where do I
> look to make my apps OS friendly?

Even if you never get the RKM's, *get the CD!* It is full of good
stuff that *any* developer for the Amiga should know. The autodocs
are available as plain text, as well as well linked amigaguide
files, linked not only to related doc files, but the include files
detailing strutures, and other nuances as well. Also a valuable
reference are the Amiga Mail articles, the publication of the old
Commodore Applications and Technical Support (CATS) group. The
later articles especially cover some topics not covered adequately
anywhere else. Plus, lots of examples (mostly SAS/C) and lots of
tools for testing and debugging. Although some parts of the CD
can be obtained elsewhere, having it all in one place (and under
one clickable index) is a real godsend.

I should note that the old CATS Developer CD 2.0 had electronic
versions of the RKM's. It was VERY cool, but short of prying it
from the still clutching fingers of a deceased developer, it is
unobtainable. As of the Amiga Developer CD V1.1, Amiga still had
not managed to obtain the rights to the electronic versions,
but hopefully that will be solved, one way or another, soon.
